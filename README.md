## 1. サービス概要

働きながらTOEIC・プログラミング・資格学習などに取り組む社会人の、<br>「努力や前進を実感しづらい」「モチベーションの維持が難しい」という課題を、<br>「学習に着手した行動や短時間でも取り組めた記録をポイント化」することで、長期的な学習継続をサポートするサービスです。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

このアイデアの原点は、自分自身が英語学習やプログラミング学習を1人でコツコツと続けてきた経験にあります。

私は、留学やワーキングホリデーをきっかけに、学習計画やTodoアプリを利用してタスク管理を行い英語学習に取り組んできました。

一人暮らしの社会人時代は、仕事や家事と並行しながら終業後や週末の隙間時間を利用して英語学習を行い、
ワーキングホリデー中は、慣れない海外生活の中で不安や孤独を感じながらも、継続的にプログラミング学習を行ってきました。

しかし、英語やプログラミングのような学習は、学習計画を立てて隙間時間で勉強しても、すぐに大きく伸びるものではありません。成果が見えづらく、前に進んでいる実感を持てないため、継続のモチベーションを保つ難しさがありました。

その中で、以下のように目標を達成できたら自分へご褒美を設定することが、学習継続のモチベーションの1つとして有効でした。

- 今週3回DMM英会話ができたら、友達とパブでビールを飲む
- 普段ならサボる時間帯に学習できたら、好きなクッキーを買う

---

### 2-2. なぜそれが気になったのか
- 自分の性格
  - 自分の成長が数字で可視化されることを楽しいと感じる
  - 完璧主義な面があり、タスクを設定してできなかった時に罪悪感や自責の念を感じる

- 過去の経験
  - 小学校から高校までは、自主的に学習に取り組む習慣がほとんどなかった
  - 学習を始めた頃はサボったり後回しにすることが多かったが、継続する中で「できなかったこと」が「できるようになる」瞬間に達成感を感じるようになった
  - 「海外で自分自身で生活基盤を築く」「ヨーロッパを1人で旅行する」という、数年前の自分からは想像もできない体験

- 大切にしている価値観
  - 特に1人で海外でも問題なく生活できるようになれたのは、コツコツと工夫しながら学習を続けた経験があったから
  - 私にとって学習とは、人生の選択肢を広げられるものだと感じている

だからこそ、学習を継続するための「工夫」や、努力を前向きに受け取れる「仕組み化すること」に価値があると考えています。

このように、「やろうと決めたことに取り組めた」「いつもならサボる時間に着手できた」などの行動を評価し、ゲームのように前向きに続けられる仕組みをアプリとして作りたいと考えたことが、このアイデアの出発点です。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

実際に想定する属性を持つ友人や同僚にヒアリングを行うことで、課題の具体化を行いました。
<br></br>
結果、働きながら学習している社会人には以下のような困りごとがあると分かりました。
```
- 学生のように、毎日同じ時間にまとまった学習時間を確保することが難しい
- 仕事の疲れから、「今日は勉強しよう」と思っていても後回しになりやすい
- 一人で学習しているため、周囲からの強制力が弱く、簡単にサボれる
- TOEIC専用アプリ、プログラミング教材、問題集、動画教材など、学習媒体が分散している
- Notion、Todoアプリ、Excel、カレンダーアプリは多機能だが、管理すること自体が負担になりやすい
- 目標を設定して学習に取り組んでも、期待したほど成果が見えず、落ち込むことがある
```
以上の内容から、より大きな問題は、忙しい中で学習に取り組んでいても、その努力や前進が実感しづらいことにあると考えました。

---

### 3-2. 本当に解決したい課題は何か

TOEIC、プログラミング、資格学習などは、成果が出るまでに時間がかかります。さらに、学習記録や達成状況が媒体ごとに分かれるため、全体としてどれだけ取り組んだかを把握しづらい状態です。

また仕事の後の疲れている状態では、
<br>
「何の勉強を」
<br>
「何分取り組むか」
<br>
などのタスク管理自体も心理負担となる可能性があります後回しにしてしまうという話もありました。

その結果、「頑張っているのに成果が出ていない」と感じてしまい、継続の意欲が下がってしまいます。

以上のことから、複雑な管理をしなくても、その日に「学習に向き合えた行動」や「小さな前進を記録・可視化」し、前向きに感じられる仕組みが重要だと考えます。

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー
- 働きながら、TOEIC・プログラミング・資格学習などのスキルアップに取り組んでいる社会人を想定

特に対象にしたいのは、教材や学習アプリはすでに使っているものの、仕事の疲れや日々の予定によって学習の着手が後回しになりやすく、「今日もできなかった」と自己嫌悪になりやすい人

年齢設定：
- 年齢：20代前半から30代前半の社会人
- 理由：キャリア初期から中堅に向かう時期で、転職・昇進・業務スキル向上のために自主学習を始める動機が生まれやすいため。また、日々の業務で疲れやすく、学習を継続する仕組みが必要になりやすいと考えています。

生活・状況設定：
- 平日は仕事や家事でまとまった学習時間を取りづらい
- 出勤前、通勤時間、昼休み、帰宅後などの短い時間で学習することが多い
- 仕事終わりは疲れており、学習内容を考えること自体が負担になりやすい
- 週末にまとまった時間を取り、平日に進められなかった分を補うことがある

環境設定：
- 基本的に一人で学習している
- TOEICアプリ、動画教材、問題集、プログラミング学習サービスなど、学習内容そのものは既存の教材で行っている
- Notion、Todoアプリ、カレンダーなどで予定管理を試したことはあるが、管理すること自体が負担になり、続かなくなることがある

このサービスでは、教材アプリや学習サービスを置き換えるのではなく「今日は学習に着手できた」「短時間でも前に進めた」という行動を記録・可視化することを目的とします。

---

### 4-2. 自分とユーザーの距離

【想定ユーザーとの距離】
<br>過去の自分、現在の自分、身近な友人
<br></br>
【理由】
<br>
私自身、英語学習やプログラミング学習に取り組む中で、仕事の疲れや他の予定を理由に学習を後回しにしてしまった経験があります。また、学習計画やTodoアプリを使っていても、予定通りにできなかった時に罪悪感が残り、継続のモチベーションが下がることがありました。身近にも、働きながら資格取得やスキルアップのために学習している人がいます。
<br></br>
そのため、「学習を始めるまでの心理的ハードル」や「短時間でも前に進めた実感の得づらさ」に焦点を当て、フィードバックを得ながらサービス設計ができると考えています。

---

### 4-3. 実際に使われる可能性について（需要・存在確認）

このアプリを思い出す場面は、以下のようなタイミングを想定しています。
```
- 仕事終わりに「今日は勉強する予定だったのに、スマホを見ていたら時間が過ぎてしまった」と感じたとき
- 学習を始めたものの、数日で途切れて自己嫌悪になったとき
- Notion、Todoアプリ、カレンダーで予定を立てても、実行直前の心理的ハードルを越えられなかったとき
- 教材アプリの正答率やスコアだけでは、自分が積み上げた努力を実感しづらいと感じたとき
```
<br></br>
今の生活の中で代わりに使われているものは、主に以下の2種類だと想定します。
```
- 専用学習アプリ：正答率、間違いの指摘、問題演習など、学習内容に対するフィードバックを行う
- タスク管理アプリ：予定、期日、タスクの書き出し、進捗管理を行う
```

ただし、これらのアプリは「疲れている中でも5分だけ始められた」「予定通りではないが、少しでも学習に向き合えた」といった行動そのものを前向きに評価する仕組みは弱いと感じています。

---

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ

このサービスは、学習内容を提供するアプリではなく、学習に取り組む行動を支えるアプリとして利用されることを想定しています。

利用の流れは以下の通りです。
```text
1. 週末や空いている時間に、ざっくり学習予定日を決める
2. アプリを開き、カレンダーUIから学習日と学習内容を入力する
3. 学習日にアプリを開き、その日の体力に合わせてタイマーを選ぶ
   選択肢: 5分、15分、30分、60分
4. 開始ボタンを押す
   実際の学習は、TOEICアプリ、動画教材、問題集、書籍などで行う
5. 学習後に完了ボタンを押す
6. 学習時間、予定通りに開始できたか、タイマーを完了できたかをもとにポイントを獲得する
7. ポイントが必要数まで貯まると、事前に自分で設定したご褒美を受け取れる
   例: 300pt到達で「週末にカフェでケーキを食べる」を達成
```
このサービスを継続して使う理由は、完璧に毎日続けることを前提にしない点にあります。

毎日長時間学習できたかではなく、ユーザーが決めた日に少しでも着手できたかを評価します。疲れている日に5分だけでも始められた行動をポイント化することで、「できなかった」ではなく「少しでも進めた」と受け取れるようにします。

将来的には、学習記録データをもとに、ユーザー自身が「どの時間帯に学習しやすいか」「どの曜日にサボりやすいか」「どのくらいの学習時間なら継続しやすいか」を確認できるようにしたいと考えています。

```text
After:
```

```text
Before:
```

## 7. このアプリで実現すること

既存の教材や学習アプリで行った学習行動を記録・評価するための仕組みとして、MVPでは以下を検証します。

```text
忙しい社会人が、短時間タイマーと行動評価、Rank・ポイント表示、ご褒美達成の演出によって、
学習に着手しその行動を前向きに受け取れるようになるか
```

---

### MVPで作る機能

#### 1. カレンダーUIから学習予定日を登録する機能

```
【課題】
- 忙しい社会人は、仕事や生活の予定によって毎日同じ時間に学習することが難しい
- その場で「今日やるかどうか」を考えると、疲れている日は後回しにしやすい

【実現したい状態】
- 事前に「この日は学習する」と決めておき、当日に迷わず学習に入れる状態にする
- 毎日継続ではなく、自分で決めた日に着手できたことを評価できるようにする
```
- カレンダーから学習予定日を選択
- 学習内容を簡単に入力
- 登録した予定日の表示

---

#### 2. タイマー機能

```
【課題】
- 仕事終わりや通勤時間では、長時間学習を前提にすると始める前の負担が大きい
- 「どれくらいやるか」を毎回考えること自体が負担になり、学習開始を後回しにしやすい

【実現したい状態】
- 長時間の学習を前提にせず、短い時間でも迷わず学習を始められる状態にする
- タイマー終了後に今回取り組んだ内容を振り返れる状態にする
```

- 5分、15分、30分、60分の固定タイマー
- タイマー開始前に「今回やること」を自由入力
  - 例：単語10個、問題5問、教材1セクション
- 開始ボタンを押すとタイマーがカウントダウンする
- タイマー終了後、または完了ボタンを押した後に、入力した「今回やること」に対する結果を選択
  - 完了できた
  - 一部取り組めた
  - 難しかったが取り組めた
- 選択した学習結果を、チャレンジポイントの判定に使う

---

#### 3. ポイント付与・Rank判定機能

```
【課題】
- 学習後の評価の指標が「達成できたか」だけでは、忙しい中で学習に向かった行動の価値を実感しにくい
- 一方で、学習時間の長さや問題の正答数で評価すると、忙しい社会人のモチベーション維持には不向き
- また、ポイントに意味がないと、単なる数字にしか感じない

【実現したい状態】
- 学習開始、選んだ時間の取り組み、予定日の実行、決めた目標への取り組みを、それぞれポイントとして扱える状態にする
- 獲得したポイントの組み合わせからRankを判定できるようにする
```

ポイントの種類：
```text
着手ポイント：
学習を開始できたことを評価する

集中ポイント：
選んだ時間だけ取り組めたことを評価する

計画ポイント：
予定日に着手できたことを評価する

チャレンジポイント：
自分で決めた小さな目標に取り組めたことを評価する
```

Rankの判定基準：
```text
C = 着手ポイントを獲得した
B = 着手ポイント + 集中ポイントを獲得した
A = 着手ポイント + 集中ポイント + チャレンジポイントを獲得した
S = 着手ポイント + 集中ポイント + チャレンジポイント + 計画ポイントを獲得した
```
Sランクは、予定日に実行できたときに発生するボーナス評価とします。<br>
そのため、予定通りにできなかった日でも、着手・集中・チャレンジの行動は評価できるようにします。

---

#### 4. 学習完了時の簡易フィードバック機能

```
【課題】
- ポイント付与とRank判定だけでは、ユーザーが「なぜそのRankなのか」「なぜそのポイントが増えたのか」を理解しづらい
- 努力の評価がブラックボックス化すると、ポイント制度の意味や達成感が伝わりにくい
- 学習後の即時フィードバックが弱いと、学習に向かった行動の価値を実感しにくい

【実現したい状態】
- 学習完了時に同じ画面内でRank、合計獲得ポイント、ポイント内訳を確認できる状態にする
- 学習直後に「何が評価されたのか」を確認できる状態にする
```

- Rankの表示
- 合計獲得ポイントの表示
- ポイント内訳の表示
- ご褒美までの進捗ポイントの表示と変化

表示例：
```text
学習完了
Rank A
+50ポイント

内訳：
```

## 8. このアプリの懸念点とその対策

【懸念点】<br>

急な残業、体調不良、家事、予定変更などによって、計画した日に必ず学習できないケースが発生する可能性があります。
<br></br>
【何が問題となるか】<br>

学習自体はできているにもかかわらず、「予定日にできなかった」という理由で**計画ポイントを得られず、Rankが低くなる**と、ユーザーが以下のように感じる可能性があります。
- 「予定通りに学習しなければならない」というストレスやプレッシャーを感じる
- ユーザー体験に柔軟性がなく、使い続けようと思えない
<br></br>

【なぜそう思うか】<br>

対象としているユーザーは、本業や生活と両立しながら学習している社会人です。完璧に計画通りの学習を求めることは、長期的な学習継続には逆効果になると考えています。

---

### 対策

MVPリリース後に、カレンダー上で作成した学習ブロックを別日に移動できる機能の実装を検討します。

```
具体例：

1. 火曜日に「英単語を15分学習」という予定を作成する
2. 当日、急な残業で学習できなくなる
3. カレンダー上の学習予定ブロックを別の曜日へ移動する
```

予定日にできなかった場合でも、学習日を簡単に変更できるようにします。無理のない日に学習を移動し、<br>柔軟な学習計画を作れるようにすることで、継続しやすくします。

## 9. 今後の展開・発展の方向性

このアプリは、**「記録して終わり」にならず、蓄積された学習データによって、自分の継続や成長を実感できるサービス**として発展させたいと考えています。

MVPでは、短時間タイマー、ポイント、Rank、ご褒美達成の演出によって、忙しい社会人が学習に着手しやすくなるかを検証します。

今後は、そこで蓄積された学習記録をもとに、ユーザーが過去の自分と比較しながら、「以前より続けられている」「前より短時間でも着手できるようになった」「自分に合う学習パターンが分かってきた」と感じられる体験に発展させたいです。
<br><br>
- ユーザーが使い続ける理由<br>
    成果がすぐに見えにくい学習でも、日々の取り組みや頑張りを数値として確認でき、
    使い続けることで、過去の自分の学習データが溜まります。その結果、進歩や成長をデータの積み重ねで実感できます。
<br><br>
- 続けることで変わること<br>
    ユーザーは自分の学習傾向を把握できるようになります。<br>
    例えば、タスクを完了させるためにどのくらい時間を使ったかのデータが溜まることで、次回の学習計画を立てるときの参考にもなります。

「気合いや自己卑下だけで学習を続ける」のではなく、「自分の生活リズムや過去の実績」に合わせて、
現実的な無理のない学習計画を立てやすくなります。
<br><br>
- データが溜まることでできること<br>
例えば、以下のようなデータを振り返れるようにします。

    - 今週学習できた日数
    - 週あたりの学習回数の自己ベスト
    - 短時間でも着手できた回数
    - 同じ種類のタスクに必要だった学習時間の目安

これらのデータを使うことで、ユーザーは「今週は2日学習できた」「以前は週1回だったが、今月は週3回学習できた」「英単語の復習には15分程度必要だった」など、過去の自分と比べた変化を確認できます。

他人と比べるのではなく、過去の自分の記録と比べることで、成果がすぐに見えにくい学習でも、継続や成長を実感しやすくなります。

---

### 他の人と関わる余地
導入候補：目標や達成期限を共有できる機能

「今月中にTOEIC単語を300個復習する」「2週間で教材を3章進める」「今週は3回、15分学習する」といった目標を友人に共有し、必要に応じて達成状況だけを見せられるようにします。

### 今考えている発展案

- 機能追加の方向性：
  - 学習実績の詳細なデータを振り返れる機能
  - ユーザーの反応を見ながら、ポイント交換型のご褒美機能を検討する

- UI・体験の改善：
  - カレンダー上で柔軟に予定を操作できるようにする
  - 予定変更があっても簡単に学習計画を変更できるようにする

- 機能以外の工夫：
  - 他人との比較ではなく、過去の自分との比較を中心にした体験設計

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- 開発言語：Ruby 4.0.x
- フレームワーク：Ruby on Rails 8.1.x
- 認証：Rails 8標準認証ジェネレーター
- フロントエンド：Hotwire（Turbo / Stimulus)
- スタイル：Tailwind CSS v4
- DB：Neon（PostgreSQL）
- 使用ライブラリ
  - カレンダーUI：simple calendar
  - テスト：RSpec
- デプロイ：Render
- インフラ：Docker / Docker Compose

---

### 10-2. 技術選定の理由

| カテゴリ    | 技術                        | 採用理由                                                                       |
| ------- | ------------------------- | -------------------------------------------------------------------------- |
| 開発言語    | Ruby 4.0.x                | Rails 8.1と組み合わせやすく、今後のRails学習でも新しい環境に慣れるため。                                |
| バックエンド  | Ruby on Rails 8.1.x       | 認証、DB操作、ルーティング、ビューなどWebアプリに必要な仕組みが揃っており、Solid Queue等を活用しインフラをシンプルに保つことができる |
| 認証      | Rails 8標準認証ジェネレーター        | MVPでは基本的なログイン機能で十分であり、Deviseよりも学習コストと実装範囲を抑えながらRails標準の認証実装を理解できるため。       |
| フロントエンド | Hotwire（Turbo / Stimulus） | Stimulusでタイマーのカウントダウンを実装し、Turboで完了後の表示更新をシンプルに行えるため。                       |
| スタイル    | Tailwind CSS v4           | ユーティリティクラスで素早くUIを組み立てられ、カレンダーやポイント表示などの画面調整をしやすいため。                        |
| データベース  | Neon（PostgreSQL）          | Render版PostgreSQLは30日の制限がある一方、Neonは使用量ベースの制限であり、MVPには十分だと判断したため。                 |
| カレンダーUI | simple calendar           | RailsでカレンダーUIを実装しやすく、学習予定日を登録・表示するMVPの主要機能に合っているため。                        |
| テスト     | RSpec                     | Railsアプリで一般的に使われているテストフレームワークで、モデルや機能の振る舞いを確認しやすいため。                       |
| デプロイ    | Render                    | Railsアプリを比較的シンプルにデプロイでき、個人開発のMVP公開に向いているため。                                |
| インフラ    | Docker / Docker Compose   | 開発環境を揃えやすく、Ruby、Rails、DBなどのバージョン差によるトラブルを減らせるため。                           |

#### 採用しない技術と理由

| 技術                | 採用しない理由                                                                                        |
| ----------------- | ---------------------------------------------------------------------------------------------- |
| Devise            | 認証機能が豊富で実績もあるが、MVPでは基本的なログイン機能で十分なため。まずはRails 8標準認証ジェネレーターを使い、Rails標準の認証実装を理解することを優先する。        |
| FullCalendar      | 高機能なカレンダーUIを実装できるが、ドラッグ操作や複雑なイベント管理はMVPでは必須ではないため。学習予定日の登録と表示を優先し、simple calendarで実装範囲を抑える。    |
| Render PostgreSQL | Renderと同じサービス内で管理できる利点はあるが、無料枠では期限の制限があるため。MVPでは継続的にDBを利用しやすいNeonを採用する。                        |
| React / Next.js   | 高度なフロントエンド開発には向いているが、MVPではRailsとHotwireで必要な画面更新やタイマー操作を実装できるため。まずはRails中心の構成にして学習コストと開発工数を抑える。 |

#### 今回チャレンジしたい点
- 学習予定、タイマー、ポイント、Rank、達成型のご褒美受け取りなどのビジネスロジックを理解し、実装する
- 「本業と両立しながら学習をしている社会人」向けに、学習の負担にならないUI/UXを設計する
- Rails 8の「外部サービスへの依存を減らす」という設計思想と、導入されたSolid Trifectaを理解・活用しながら開発を行う

#### 不安な点
- テーブル設計が複雑になるためER図の作成に時間がかかると予測される
- モデル間の正しいアソシエーション理解にも時間がかかると予測される
- Rails 8系は比較的新しいバージョンで、Rails 7系とは異なる点があるため、バージョン差による環境構築やアプリ開発で苦戦する可能性がある
